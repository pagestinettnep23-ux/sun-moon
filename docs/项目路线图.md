# 项目路线图

## 1. 项目定位

本项目计划基于 Uniswap v4 Hooks 开发一个 Web3 应用，先完成智能合约原型、测试、前端交互和文档体系，后续根据生态支持情况评估部署到 BNB Chain 或其他兼容 EVM 的网络。

当前阶段的核心目标不是马上写复杂合约，而是先明确产品方向、技术路线、安全边界和开发顺序，避免后面一边写代码一边大幅返工。

## 2. 建议 MVP 方向

初版建议先做一个“带手续费管理和基础风控能力的 Uniswap v4 Hook”。

MVP 可以包含：

- Hook 在指定池子交易前后执行自定义逻辑。
- 支持项目方配置基础参数，例如手续费开关、白名单、最大单笔限制等。
- 前端展示 Hook 状态、池子信息和关键配置。
- 测试覆盖核心路径，包括正常交易、非法参数、权限控制和边界场景。

暂不建议初版就做：

- 复杂动态费率算法。
- 高风险自动做市策略。
- 跨链逻辑。
- 复杂代币经济模型。
- 未经充分测试的收益分配机制。

## 3. 阶段规划

### 阶段 0：项目准备

目标：搭建清晰的项目结构和规划文档。

任务：

- 梳理项目需求、技术方案、安全检查清单。
- 确认使用 Foundry 还是 Hardhat 作为合约开发工具。
- 确认前端技术栈，建议使用 React + TypeScript + Vite。
- 确认后续是否需要部署到测试网。

交付物：

- `docs/项目路线图.md`
- `docs/需求说明.md`
- `docs/技术方案.md`
- `docs/安全检查清单.md`

### 阶段 1：合约开发环境

目标：让项目可以编译、测试和本地运行。

任务：

- 初始化智能合约开发框架。
- 引入 Uniswap v4 相关依赖。
- 配置 Solidity 版本、测试环境、格式化工具。
- 写一个最小 Hook 合约骨架。
- 写第一个基础测试，确认环境可用。

交付物：

- 可编译的合约工程。
- 可运行的测试命令。
- 最小 Hook 合约。

### 阶段 2：Hook MVP 功能

目标：实现第一版可验证业务逻辑。

任务：

- 实现 Hook 权限和参数配置。
- 实现交易前或交易后的基础限制逻辑。
- 实现必要事件，方便前端和区块浏览器追踪。
- 完成核心单元测试。

交付物：

- MVP Hook 合约。
- 单元测试。
- 合约接口说明。

### 阶段 3：前端 MVP

目标：让普通用户可以通过网页理解并操作项目。

任务：

- 初始化前端项目。
- 连接钱包。
- 展示网络、账户、Hook 配置和池子状态。
- 提供管理员配置页面。
- 提供用户查看页面。

交付物：

- 可本地运行的前端。
- 基础钱包连接能力。
- 基础合约读取和写入能力。

### 阶段 4：测试网部署

目标：在测试网验证真实链上流程。

任务：

- 选择测试网络。
- 准备部署脚本。
- 部署 Hook 合约。
- 记录部署地址和参数。
- 通过前端连接测试网验证流程。

交付物：

- 测试网部署记录。
- 可访问测试网合约的前端配置。
- 测试报告。

### 阶段 5：安全加固

目标：降低资金和权限风险。

任务：

- 补充边界测试、模糊测试和异常路径测试。
- 检查权限控制。
- 检查重入、价格操纵、参数滥用等风险。
- 准备外部审计或社区 review 材料。

交付物：

- 安全检查记录。
- 完整测试报告。
- 审计准备文档。

### 阶段 6：BNB Chain 可行性评估

目标：确认是否适合部署到 BNB Chain。

任务：

- 确认 BNB Chain 对 Uniswap v4 相关基础设施的支持情况。
- 确认 PoolManager、PositionManager、Universal Router 等依赖是否可用。
- 评估部署成本、用户来源和生态支持。
- 如果 BNB Chain 暂不适合，准备备选网络方案。

交付物：

- BNB Chain 部署评估报告。
- 主网部署前检查清单。

## 4. 当前最近任务

接下来建议按这个顺序推进：

1. 完成项目规划文档。
2. 初始化合约开发环境。
3. 写最小 Hook 合约。
4. 写测试。
5. 初始化前端。
6. 做本地演示。

## 5. 项目原则

- 先小后大：先做可运行的 MVP，再逐步增强。
- 先测试后上链：没有测试覆盖的合约不部署。
- 先测试网后主网：所有核心流程必须先在测试网验证。
- 先安全后收益：涉及资金的逻辑以安全为第一优先级。
- 文档同步更新：每次重要决策都记录到 docs 中。
