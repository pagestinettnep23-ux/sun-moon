## 官方参考

- Base 官方文档记录：Base Mainnet chainId 是 `8453`，公共 RPC 是 `https://mainnet.base.org`；Base Sepolia chainId 是 `84532`，RPC 是 `https://sepolia.base.org`。Base 官方也提示公共 endpoint 有限流，不适合作为生产系统的唯一依赖。参考：[Base Connecting to Base](https://docs.base.org/chain/using-base)、[Base RPC Overview](https://docs.base.org/base-chain/api-reference/rpc-overview)。
- Safe 官方文档说明：Safe Smart Account 支持多个 owner 和 threshold，多签交易需要达到 threshold 后才能执行。参考：[Safe Smart Account overview](https://docs.safe.global/advanced/smart-account-overview)。
- Safe 帮助文档建议 threshold 高于 `1`，避免单个账户独自执行交易；常见个人方案是 3 个 signer、2 个确认。参考：[What Safe setup should I use?](https://help.safe.global/articles/1038062742-what-safe-setup-should-i-use)。

## 一句话结论

主网不应该复用任何 Base Sepolia 测试钱包。最新目标是上线后只保留协议经费收款地址，其他管理权限全部放弃。2026-05-16 新方向确认：SUN/MOON 都保持自由转账，不再试图从 token 合约层禁止市场自行建池；项目只对明确支持的 v4 Hook 池收费。`MAINNET_ADMIN_SAFE` 如果存在，也只应作为上线配置期的临时管理员；配置完成后必须逐项 renounce 或进入等效不可管理状态。

## 推荐主网结构

```text
MAINNET_DEPLOYER
  - 新建，只放少量 ETH 用于部署 gas
  - 不做 owner，不做长期管理员，不接收协议收入
  - 部署结束后余额转走或停用

MAINNET_ADMIN_SAFE
  - 推荐 2-of-3；团队扩大后可升级到 3-of-5
  - 只作为 SUN/MOON token、curve、Hook、adapter、CREATE2 deployer 的上线配置期 owner
  - 负责 setMinter、setMoonCurve、setMoonAMM、setAllowedMoonPool、setSwapAdapter、setProtocolBudget、setPaused
  - 所有配置完成并复核后，逐项执行 renounceOwnership 或等效放弃管理权

PROTOCOL_BUDGET_SAFE
  - 推荐 2-of-3；不要和部署钱包相同
  - 作为 protocolBudget / fee recipient
  - 只收款和财务管理，不参与日常合约配置，不做 owner
  - 不做 deployer、adapter owner、pause guardian、CREATE2 owner
  - 不参与任何合约配置投票

SUPPORTED V4 HOOK POOLS
  - SUN/MOON 都保持自由转账
  - 市场自行创建的 AMM 池不由协议承诺或控制
  - 项目支持 SUN/USDC v4 Hook 池，swap 收 2% USDC
  - 项目支持 MOON/USDC v4 Hook 池，swap 收 5% USDC
  - 第三方池不代表协议价格，也不保证触发费用回灌
```

## 为什么要分开

- 部署钱包拿过私钥、跑过命令，暴露面最大，不应该长期控制合约。
- 管理权限能改白名单、adapter、budget、暂停状态，必须多人确认。
- 预算钱包接收协议费用，属于财务角色，不能和操作权限混在一起。
- 前端配置错误可能把用户带到错误合约，所以前端地址也要双人复核。

## 当前合约对应的权限点

从当前代码看，以下操作属于 owner 权限，主网上线配置期可由临时 `MAINNET_ADMIN_SAFE` 控制；配置完成后应全部放弃：

| 合约/模块 | 权限函数 | 风险说明 |
| --- | --- | --- |
| `SunToken` | `setMinter()` | 影响 SUN 铸造权限 |
| `MoonToken` | `setMinter()` | 影响 MOON 铸造权限 |
| `SunCurve` | `setMoonCurve()`、`setMoonAMM()` | 影响 SUN/MOON 曲线联动和 AMM 费用入口 |
| `MoonCurve` | owner 权限 | 影响 MOON 曲线配置和治理 |
| `BaseMoonAmmFeeV4Hook` | `setProtocolBudget()`、`setSwapAdapter()`、`setAllowedMoonPool()`、`setPaused()` | 影响协议收款、adapter、池白名单和暂停 |
| `TestnetUsdcAdapter` / 正式 adapter | `setAuthorizedHook()`、`setPaused()`、router/token route allowlist | 影响 Hook 授权和手续费兑换路径 |
| `Create2HookDeployer` | `predictHook()`、`deployHook()` 的 owner 限制 | 影响 Hook 是否能按预测地址部署 |

重要提醒：`TestnetUsdcAdapter` 是测试网适配器，不应默认作为主网正式 adapter。主网 adapter 方案必须另写并审计。

新增主网目标：上线后只保留 `PROTOCOL_BUDGET_SAFE` 收款地址。`PROTOCOL_BUDGET_SAFE` 的唯一权益是接收协议经费，不得被赋予 owner、pause、adapter、CREATE2、白名单、部署或任何治理权限。SUN/MOON 自由转账，项目支持 `SUN/USDC` 和 `MOON/USDC` 两个 v4 Hook 池收费；第三方池不代表协议支持路径。`BaseMoonAmmFeeV4Hook` 当前 owner 是 immutable，不能 renounce；这与新目标冲突，主网前必须先改为支持放弃 owner 的统一 Hook 或采用等效不可管理设计。详见 `docs/Base-主网上线后最小权限与放弃管理权方案-2026-05-15.md` 和 `docs/Base-SUN-MOON-v4-Hook池新方案-2026-05-16.md`。

## 多签 signer 设计

推荐起步方案：

```text
MAINNET_ADMIN_SAFE:
  owners=3
  threshold=2

PROTOCOL_BUDGET_SAFE:
  owners=3
  threshold=2
```

3 个 signer 的建议分布：

| signer | 建议 | 不建议 |
| --- | --- | --- |
| Signer A | 项目 owner 的硬件钱包 | 浏览器热钱包单独控制 |
| Signer B | 第二个硬件钱包或可信共同管理员 | 和 Signer A 同一助记词派生 |
| Signer C | 备用恢复 signer，离线保管 | 交易所充值地址、托管账户、临时钱包 |

禁止事项：

- 不要用同一套助记词派生 3 个 signer。
- 不要把 signer 私钥存在聊天、网盘、截图、浏览器收藏、代码仓库。
- 不要用 Base Sepolia 测试钱包当主网 signer。
- 不要把部署钱包加入预算 Safe 作为唯一关键 signer。

## 是否需要快速暂停权限

当前合约里的暂停函数是 owner 权限：

```text
BaseMoonAmmFeeV4Hook.setPaused(bool)
TestnetUsdcAdapter.setPaused(bool)
```

如果 owner 是 Safe，多签确认后才能暂停。这更安全，但紧急情况下可能不够快。最新目标是上线后放弃管理权，所以暂停权限也不能长期保留。

主网前需要重新选择：

| 方案 | 做法 | 优点 | 风险 |
| --- | --- | --- | --- |
| A. 上线后放弃暂停权限 | 配置期可暂停；正式开放前 renounce owner | 最符合无管理员目标 | 上线后无法紧急暂停 |
| B. 保留多签暂停权限 | 不 renounce 或保留单独 pause guardian | 紧急响应能力更强 | 不符合“其他全部放弃管理权限” |
| C. 只保留一次性迁移/暂停机制 | 单独设计更窄权限并审计 | 折中 | 需要改代码、测试和审计 |

当前按用户目标选择方案 A：上线后放弃暂停和恢复权限。这个决定风险更高，但权限最小；主网前必须把风险写进用户提示和审计清单。

填写要求：

- 全部必须是公开地址。
- 不允许填写私钥、助记词、恢复词。
- 每个地址必须说明用途。
- `PROTOCOL_BUDGET_SAFE` 只接收协议经费，不允许兼任任何管理员或部署角色。
- `MAINNET_ADMIN_SAFE` 和 `PROTOCOL_BUDGET_SAFE` 不建议相同。
- `MAINNET_DEPLOYER` 不应等于任何 Safe signer 的唯一控制地址。
- 不再填写 `SUN_FIRST_LIQUIDITY_PROVIDER`；新方向不再采用“禁止加池 / 首池守卫”模型。
- 更完整的正式参数表见 `docs/Base-主网正式参数确认清单模板-2026-05-16.md`；该表也只允许填写公开地址和公开参数。

## 主网前准备顺序

1. 创建新的主网 signer，优先硬件钱包。
2. 在 Base Mainnet 创建 `MAINNET_ADMIN_SAFE`，先小额测试签名流程。
3. 创建 `PROTOCOL_BUDGET_SAFE`，先小额测试收款和转账流程；它只收协议经费，不参与任何管理。
4. 确认 Safe owners 和 threshold，并截图或导出公开配置记录，但不包含私钥。
5. 把 Safe 地址填入主网参数草案。
6. 在本地脚本里加入 Base Mainnet chainId 保护、人工确认变量和地址非零检查。
7. 做 mainnet fork dry-run，只读和模拟，不广播。
8. 外部 review / 审计通过后，才允许准备主网广播草案。

## 主网部署时的 owner 原则

优先原则：

```text
能在 constructor 里直接设置 owner = MAINNET_ADMIN_SAFE，就不要先给 EOA。
```

如果脚本因为部署流程必须先由 `MAINNET_DEPLOYER` 临时拥有：

- 必须在同一份部署计划里列出 `transferOwnership(MAINNET_ADMIN_SAFE)`。
- transfer 前不允许注入真实流动性。
- transfer 后必须用链上读数复核 `owner()==MAINNET_ADMIN_SAFE`。
- 如果任何 owner 没有转给 Safe，停止后续所有主网动作。
