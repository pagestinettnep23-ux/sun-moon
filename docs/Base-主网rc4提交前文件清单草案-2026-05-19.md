# Base 主网 rc4 提交前文件清单草案 - 2026-05-19

本文给技术小白使用，目的只有一个：在创建 `rc4` 审计候选版本之前，先确认哪些文件应该进入本次合约审计提交，哪些文件暂时不要进入。

这不是主网部署批准，也不是广播记录。

```text
不执行 Base 主网交易
不使用真实资金
不要求或记录私钥、助记词、恢复词
不使用 --private-key
不提交
不打 tag
```

## 1. 本次 rc4 的定位

`rc4` 是合约审计候选版本，不是前端发布版本。

本次 `rc4` 应该包含：

```text
合约修复
对应测试
主网 CREATE2 / poolId / dry-run 文档
上线前人工复核清单
审计修复记录和审计输入包索引
Base Sepolia 3 天只读观察资料
```

本次 `rc4` 暂时不包含：

```text
frontend/
```

原因：

```text
frontend 是前端展示和本地预览资料。
这次 rc4 的重点是合约安全审计。
把前端混进来会让审计范围变杂。
```

## 2. 必须进入 rc4 的合约文件

```text
contracts/SunCurve.sol
contracts/MoonCurve.sol
contracts/hooks/base/BaseSunMoonUsdcFeeV4Hook.sol
```

为什么：

```text
SunCurve / MoonCurve 修复了 lastMintBlock DoS。
BaseSunMoonUsdcFeeV4Hook 修复了空 hookData 兼容性。
这些是本轮审计必须看的核心改动。
```

## 3. 必须进入 rc4 的测试文件

```text
test/SunCurve.t.sol
test/MoonCurve.t.sol
test/hooks/base/BaseSunMoonUsdcFeeV4Hook.t.sol
```

为什么：

```text
这些测试直接证明：
同一付款地址同区块 mint 后 burn 仍会失败。
不同用户不会再互相阻塞 burn。
空 hookData 的正常 swap 可以成功。
min=0、min 过高、极小额 fee=0 仍然失败。
```

## 8. 不要使用的做法

提交 rc4 前，不建议使用：

```text
git add .
git add -A
```

原因：

```text
这两个命令会把 frontend/ 也一起加入暂存区。
本轮 rc4 不建议包含 frontend/。
```
