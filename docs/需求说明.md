# 需求说明

## 1. 项目背景

Uniswap v4 引入 Hooks 机制，允许开发者在池子的关键生命周期节点插入自定义逻辑，例如初始化池子、添加流动性、移除流动性、交易前、交易后等。

本项目希望围绕 Uniswap v4 Hooks 做一个可落地的 Web3 应用，覆盖智能合约、测试、前端和部署流程。用户当前偏技术入门，因此项目需要保持清晰、可解释、可测试，并避免一开始就引入过高复杂度。

## 2. 项目目标

### 业务目标

- 做出一个基于 Uniswap v4 Hook 的 MVP。
- 支持用户通过前端查看 Hook 状态和相关池子信息。
- 支持管理员通过前端配置 Hook 的基础参数。
- 为后续部署到 BNB Chain 或其他 EVM 网络预留空间。

### 学习目标

- 理解智能合约项目的基本结构。
- 理解 Uniswap v4 Hooks 的基本工作方式。
- 学会通过测试验证合约行为。
- 学会用前端连接钱包并读取链上数据。

### 工程目标

- 合约代码清晰、测试充分。
- 前端操作简单，适合非技术用户理解。
- 所有关键操作有文档说明。
- 部署流程可复现。

## 3. MVP 功能范围

### 智能合约功能

第一版 Hook 建议包含以下能力：

- 管理员权限：只有管理员可以修改核心配置。
- 参数配置：可配置是否启用某些限制或费用逻辑。
- 交易检查：在交易前或交易后执行简单校验。
- 事件记录：关键配置变更和 Hook 执行结果需要发出事件。
- 只针对指定池子生效：避免 Hook 被错误地用于不支持的池子。

可选功能：

- 白名单用户。
- 黑名单用户。
- 最大单笔交易限制。
- 简单固定附加手续费。
- 暂停开关。

### 前端功能

第一版前端建议包含：

- 钱包连接。
- 当前网络显示。
- 当前账户显示。
- Hook 合约地址显示。
- Hook 配置读取。
- 管理员配置入口。
- 基础错误提示。

管理员页面建议包含：

- 修改开关类配置。
- 修改数值类配置。
- 查看最近操作结果。
- 防止用户在错误网络操作。

普通用户页面建议包含：

- 查看 Hook 是否启用。
- 查看主要规则说明。
- 查看相关池子或代币信息。

## 4. 非 MVP 范围

以下内容暂不放入初版：

- 主网部署。
- 真金白银资金管理。
- 复杂收益分配。
- 复杂动态费率算法。
- 跨链桥。
- 治理代币。
- DAO 投票。
- 复杂 K 线和数据分析系统。

这些功能可以等 MVP 稳定后再评估。

## 5. 用户角色

### 项目管理员

拥有配置 Hook 参数的权限，需要能够通过前端完成常见配置操作。

典型需求：

- 查看当前 Hook 状态。
- 修改基础参数。
- 暂停或恢复 Hook。
- 确认交易是否成功。

### 普通用户

通过前端了解项目规则，未来可能通过去中心化交易入口与池子交互。

典型需求：

- 看懂项目规则。
- 知道当前网络是否正确。
- 知道 Hook 是否启用。
- 能看到必要的风险提示。

### 开发者

维护合约、测试和前端。

典型需求：

- 快速运行测试。
- 快速部署测试合约。
- 快速定位问题。
- 清楚每个配置项的作用。

## 6. 成功标准

MVP 成功的标准：

- 合约可以本地编译。
- 核心 Hook 行为有测试覆盖。
- 前端可以连接钱包并读取合约状态。
- 管理员可以通过前端完成至少一个配置操作。
- 文档能让新手理解项目结构、运行方式和风险点。

## 7. 约束条件

- 不在没有充分测试的情况下部署主网。
- 不把私钥写入代码或提交到仓库。
- 不直接处理真实用户资金。
- 不承诺收益。
- 不使用不明来源的合约依赖。

## 8. 待确认问题

- 第一版 Hook 的具体业务主题：手续费管理、风控限制、白名单池子、积分记录，还是其他方向。
- 目标测试网络。
- 是否必须部署到 BNB Chain，还是先用 Uniswap v4 官方支持更完善的测试环境。
- 是否需要后端服务。
- 前端是否只做管理后台，还是也做面向普通用户的展示页。
