## 5. 已知问题和当前结论

| 编号 | 当前状态 | 当前结论 | 希望外部 AI 做什么 |
|---|---|---|---|
| H-02 afterSwap delta sign | disputed / not confirmed | 暂未复现双重收费 | 如果认为是 High，请给 exact path / failing assertion |
| C-1 USDC/USDT 命名错配 | config risk | 不是当前已部署漏洞，但上线前必须硬检查 | 检查是否还有遗漏的地址错配防线 |
| LMB-01 lastMintBlock DoS | fixed | 已改为 mapping(address => uint256) | 检查是否仍有跨用户同区块 DoS |
| HD-01 空 hookData 兼容性 | fixed | 空 hookData 默认 minUSDCToSunCurve=1 | 检查兼容性是否引入新绕过 |

小白解释：

```text
H-02：AI 曾担心 Hook 收费方向错。我们补了 exact-output 测试，目前没复现双重收费。
C-1：代码里有 usdt 命名，但实际主网应传 Base 官方 USDC。未来填错地址会很严重，所以必须硬检查。
LMB-01：过去一个用户 mint 可能卡住另一个用户同区块 burn。现在按地址记录，测试已覆盖。
HD-01：过去空 hookData 会失败。现在空 hookData 可以走默认保护值，但极小额和无效数据仍会失败。
```

## 8. 外部 AI 输出后怎么处理

拿到外部 AI 输出后，不要直接改代码。先按下面规则登记：

```text
Critical / High：
必须有可复现路径或失败断言。
没有复现路径，先标记 needs reproduction。

Medium / Low：
登记到 AI 审计问题跟踪表，再判断是否需要测试或文档硬闸门。

Info / 命名建议：
不要立刻改生产合约，先判断是否会改变字节码、salt、PREDICTED_HOOK、poolId。
```

登记文件：

```text
docs/Base-主网AI审计问题跟踪表-2026-05-20.md
```

当前原则：

```text
不能有 open 的 High / Critical。
不能有 needs reproduction 的 High / Critical。
所有 fixed 都必须有测试或复核记录。
所有 config risk 都必须进入上线前人工硬闸门。
```
