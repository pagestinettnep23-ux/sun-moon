# 技术方案

## 1. 总体架构

项目建议分为四层：

- 智能合约层：实现 Uniswap v4 Hook 逻辑。
- 测试层：验证合约行为、安全边界和部署脚本。
- 前端层：连接钱包，读取合约状态，提交管理员交易。
- 文档层：记录需求、路线图、技术决策和安全检查。

当前项目目录建议保持如下结构：

```text
uniswap-v4-hook-project/
  contracts/        智能合约
  test/             合约测试
  script/           部署和辅助脚本
  frontend/         前端应用
  docs/             项目文档
  README.md         项目入口说明
```

## 2. 技术栈建议

### 智能合约

建议使用：

- Solidity：智能合约语言。
- Foundry：合约开发、测试和部署工具。
- Uniswap v4 core：Uniswap v4 核心合约依赖。
- Uniswap v4 periphery：Hook 开发和周边工具依赖。
- OpenZeppelin Contracts：常用安全组件，例如权限控制、暂停开关。

选择 Foundry 的原因：

- 测试速度快。
- 适合 Solidity 原生测试。
- Web3 合约项目中使用广泛。
- 后续做模糊测试和安全测试更方便。

### 前端

建议使用：

- React：构建用户界面。
- TypeScript：减少低级错误。
- Vite：轻量、启动快。
- wagmi：连接钱包和合约交互。
- viem：底层 EVM 数据读写。
- Tailwind CSS：快速搭建界面样式。

### 网络和部署

开发阶段建议顺序：

1. 本地链。
2. Uniswap v4 支持较完整的测试网。
3. 评估 BNB Chain 测试网。
4. 再决定是否部署主网。

BNB Chain 部署需要先确认：

- 是否已有 Uniswap v4 关键合约部署。
- 对应网络的 PoolManager 地址。
- Hook 地址挖矿或权限位要求是否能满足。
- 前端钱包和 RPC 是否稳定。

## 3. Uniswap v4 Hook 基础概念

Uniswap v4 Hook 可以在池子的生命周期中执行自定义逻辑。常见 Hook 点包括：

- `beforeInitialize`
- `afterInitialize`
- `beforeAddLiquidity`
- `afterAddLiquidity`
- `beforeRemoveLiquidity`
- `afterRemoveLiquidity`
- `beforeSwap`
- `afterSwap`
- `beforeDonate`
- `afterDonate`

每个 Hook 地址需要与启用的权限匹配。也就是说，Hook 合约地址本身会编码它支持哪些 Hook 回调权限。因此部署 Hook 时可能需要特殊流程来得到符合要求的地址。

## 4. MVP 合约设计

建议初版合约命名为：

```text
ManagedHook.sol
```

建议核心能力：

- `owner`：管理员。
- `poolEnabled`：指定池子是否启用。
- `paused`：暂停开关。
- `maxSwapAmount`：最大单笔交易限制，可选。
- `allowlist`：白名单，可选。
- `beforeSwap` 或 `afterSwap`：执行核心检查逻辑。

建议事件：

- `PoolEnabledSet`
- `PausedSet`
- `MaxSwapAmountSet`
- `AllowlistSet`
- `SwapChecked`

建议错误：

- `NotAuthorized`
- `PoolNotEnabled`
- `HookPaused`
- `SwapAmountTooLarge`
- `AddressNotAllowed`
- `InvalidParameter`

## 5. 前端设计

前端建议分成三个区域：

- 顶部状态栏：钱包、网络、账户。
- Hook 状态面板：合约地址、暂停状态、主要参数。
- 管理员操作面板：修改参数、提交交易、显示结果。

页面不需要做成营销网站，第一版更像一个清晰的管理工具。

建议页面：

- `/`：项目状态总览。
- `/admin`：管理员配置页面。
- `/docs`：简单规则说明，可选。

## 6. 测试策略

合约测试优先级：

1. 部署成功。
2. 初始化参数正确。
3. 管理员可以修改配置。
4. 非管理员不能修改配置。
5. Hook 在正常场景下允许交易。
6. Hook 在暂停场景下阻止交易。
7. Hook 对超过限制的交易进行处理。
8. 事件是否正确发出。

后续增强：

- 模糊测试。
- 边界值测试。
- Gas 成本统计。
- 分叉测试。
- 测试网端到端验证。

## 7. 配置和密钥管理

必须遵守：

- 私钥只放在本地 `.env` 文件或安全的钱包工具中。
- `.env` 必须加入 `.gitignore`。
- 不在文档、代码、截图中暴露私钥和助记词。
- RPC URL 和区块浏览器 API Key 可以使用环境变量管理。

建议环境变量：

```text
PRIVATE_KEY=
RPC_URL=
BNB_TESTNET_RPC_URL=
ETHERSCAN_API_KEY=
BSCSCAN_API_KEY=
```

## 9. 关键技术风险

### Hook 地址权限风险

Uniswap v4 Hook 权限与合约地址相关，如果部署地址不匹配，Hook 可能无法正常工作。

应对方式：

- 使用官方或社区成熟的 Hook 部署工具。
- 在测试中验证 Hook 权限。
- 部署前记录权限位和合约地址。

### 网络兼容风险

不同 EVM 网络对 Uniswap v4 的支持情况不同，BNB Chain 是否适合部署需要实测和确认。

应对方式：

- 先在支持更明确的测试环境开发。
- 再做 BNB Chain 可行性验证。

### 安全风险

Hook 可以影响交易行为，一旦逻辑错误，可能导致交易失败、资金损失或被攻击。

应对方式：

- 限制初版功能范围。
- 保持逻辑简单。
- 测试覆盖异常路径。
- 主网前进行外部 review 或审计。

## 10. 阶段 2 Hook/AMM 接入原则

阶段 2 的合约侧目标是为 Hook/AMM 做准备，不是直接进入生产级 AMM。

当前原则：

- SUN/MOON 的核心入口仍是站内曲线 Mint/Burn。
- AMM 只作为辅助层，用于 MOON/USDC 流动性和 MOON 交易税费。
- SUN 官方永久不做 AMM 流动性；早期 SUN 加池守卫仅作为历史原型保留。
- MOON AMM 税费先做 Mock 测试：5% 费用中，3% 换成 USDT/USDC 后进入 SUN 飞轮路径，2% 保留原手续费资产进入协议预算钱包。
- 外部换币必须通过适配器隔离，并强制 `minUSDTOut`。
- Uniswap v4 与 PancakeSwap Infinity 的 Hook 权限模型不同，部署和测试文档必须分开。
